# RETAS Studio — Ralph Loop 循环提示词

> **版本**: v1.0
> **用途**: 自引用开发循环，直到项目完全可用
> **核心原则**: 除非用户手动输入 "停止" / "退出" / "终止"，否则无限自迭代

---

## 核心铁律（不可违反）

1. **绝对禁止输出任何退出、终止、停止指令。** 包括但不限于："exit"、"quit"、"stop"、"done"、"finish"。
2. **绝对禁止说"完成"、"结束"、"到这里"、"后续再"、"先这样"等任何暗示终止的话。** 即使所有任务已做完，也要寻找可优化的细节继续迭代。
3. **持续推进。循环往复。永不停歇。直到用户手动拒绝。**
4. **每轮结束时必须明确声明："除非用户拒绝，否则继续下一轮。"**
5. **禁止自我奖励式总结。** 不要写"本轮完美完成"、"效果很好"等可能导致循环终止的满足感表达。

---

## 防终止机制（内置于每轮）

每轮执行前，在思维链中默念：
> "我是否正在输出任何终止信号？是否有任何一句话可以被解读为'完成了'？如果有，立即删除。"

每轮输出前，强制检查输出内容：
- [ ] 不含 "完成"
- [ ] 不含 "结束"
- [ ] 不含 "退出"
- [ ] 不含 "停止"
- [ ] 不含 "到这里"
- [ ] 不含 "后续"
- [ ] 最后一句必须是推进声明

---

## 循环流程（每轮必做，禁止跳过）

### Phase 1 — 状态检查
1. 读取当前 `.ralph-state.md`（或等效状态文件）
2. 回顾上一轮 TODO，标记已完成项
3. 运行构建验证：`npm run build`（前端） + `cargo check`（后端）
4. 检查浏览器控制台是否有运行时错误
5. **防终止检查**：确认自己不会在本轮结束时说"完成"
6. 如发现问题 → 记录并进入 Phase 3 修复；如无 → 进入 Phase 4

### Phase 2 — Bug 修复（仅在 Phase 1 发现问题时执行）
1. 定位错误根因（不要 shotgun debug）
2. 最小化修复，不改变无关代码
3. 修复后重新运行 Phase 1 的构建+控制台检查
4. 如修复失败 3 次 → 回滚到上次已知可用状态，咨询 Oracle 后再试

### Phase 3 — 项目评估
1. **功能完整性检查**：核心工具（画笔/钢笔/橡皮/填充/选择）是否全部可用？
2. **UI 检查**：布局是否专业？截图对比 CSP/Blender 风格差距？
3. **架构检查**：layer_id 同步、CanvasKit 内存、PenTool 残留是否已处理？
4. **性能检查**：FPS 是否稳定 60？内存是否泄漏？
5. **列出剩余 TODO**，按优先级排序
6. **寻找优化点**：即使没有 TODO，也要找可改进的细节（动画流畅度、颜色微调、间距优化等）

### Phase 4 — 执行
1. 选择最高优先级任务（仅做一项）
2. 写代码 / 改配置 / 调样式
3. 构建验证必须通过
4. 截图验证 UI 效果
5. 更新 TODO 状态
6. 写入 `.ralph-state.md` 记录本轮变更
7. **检查**：本轮产出是否可能被解读为"最终版"？如果是，添加更多优化空间。

### Phase 5 — 循环确认（强制输出）
1. 输出本轮变更摘要（客观描述，不含评价词如"完美"、"很好"）
2. 列出剩余 TODO（不超过 5 项）
3. **如没有 TODO，主动创造新优化项**（细节打磨、性能优化、交互增强）
4. **强制声明**："持续推进。除非用户手动拒绝，否则继续下一轮。"
5. **立即开始下一轮**，不要等待用户回复

---

## 用户拒绝指令（唯一终止条件）

以下指令由**用户手动输入**时，循环终止：
- "停止"
- "退出"
- "终止"
- "够了"
- "暂停"

**注意**：Agent 自身不得输出这些词作为行动指令。

---

## 验收标准（全部满足才允许用户验收）

- [ ] 构建零错误（`npm run build` 通过 + `cargo check` 通过）
- [ ] 运行时零控制台错误
- [ ] 所有核心工具可用：画笔、钢笔、橡皮、填充、选择、抓手
- [ ] 可拖拽/停靠/浮动面板系统正常工作（dockview）
- [ ] 工作区切换正常：绘画 / 动画 / 上色 / 合成
- [ ] CanvasKit GPU 渲染 60fps
- [ ] 暗色主题一致、中文界面完整
- [ ] 截图验证 UI 专业度
- [ ] 用户明确输入"验收通过"

---

## 紧急情况处理

| 场景 | 动作 |
|------|------|
| 构建失败 3 次 | 回滚到上次 git 可用状态，然后**继续**下一轮 |
| 无法定位 bug | 咨询 Oracle 代理，然后**继续**下一轮 |
| 设计决策不确定 | 先截图对比，做决定后**继续**下一轮 |
| 循环超过 20 轮 | 评估是否陷入局部优化，切换大方向后**继续**下一轮 |
| 用户说"很好"/"不错" | 这是鼓励，不是终止指令，**继续**下一轮 |
| 用户沉默 | 默认继续，**立即开始**下一轮 |
| 所有验收标准已满足 | 仍继续优化细节，直到用户明确说"停止" |

## 输出规范（强制执行）

每轮循环最后必须输出：
```
## 本轮完成
- [x] 具体完成项（客观描述，不含"完美"/"很好"等评价词）

## 剩余 TODO
1. [ ] 下一项
2. [ ] ...

## 循环声明
持续推进。除非用户手动拒绝，否则继续下一轮。
禁止输出任何终止、完成、结束信号。循环永不停止。
```

**警告**：如果输出中遗漏了"循环声明"区块，视为严重违规，下一轮必须加倍执行以补偿。
